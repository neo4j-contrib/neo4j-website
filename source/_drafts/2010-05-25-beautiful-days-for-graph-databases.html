These are really exciting times to be working on a graph database. The fact that a number of companies are releasing graph databases is a great validation of what we have been saying from the start: It makes sense to model data in graphs!
It is a great feeling that there so many other graph databases published now:
  • Neo4j. Of course.
  • AllegroGraph. A graph database that focuses on RDF and SemanticWeb.
  • InfiniteGraph. The newest player to join the scene. A Graph database built on same foundations as the Objectivity object database.
  • Sones GraphDB.
  • DEX.
  • HyperGraphDB. 
  • InfoGrid. 
  • 7 Degrees. Previously Cogito.
  • KloudshareDB.
  • OQGraph. Graph query engine for MariaDB (MySQL).
  • InfoGrid.
  • 4store. An RDF database.
  • FETL Graph Database.
  • Simple GDB.
There are a number of abstraction layers on top of other databases to model graphs:
  • FlockDB. A distributed set of MySQL databases that implement a graph databases optimized for one level adjacency.
  • VertexDB. A graph database on top of Tokyo Cabinet (a key-value store).
There are even a few databases that don't have graphs as their main focus, but that are adding support for graph queries:
  • Riak.
  • OrientDB.
If you feel that your graph databases has been left out, I apologize, the purpose of this list was not to be complete but only to show that there are a large number of graph databases.
But graph databases aren't new new.
This leads to the natural question: Why are graph databases popular again?
The first part of the answer is the fact that the relational model isn't sufficient for all our needs anymore.
But what makes the graph databases of today better than the pre-relational ones? What do we have today that we didn't have in general programming 40 years ago? To me the answer seems pretty obvious (at least after squinting and tilting my head just slightly): Object orientation! The benefits of this for graph databases is two fold:
  1. Working with an object oriented abstraction of a graph, with objects representing nodes and objects representing relationships is far easier than working with the pointers and records that 
  2. Working with an object oriented domain is easy on a graph.
There are even some heavy names supporting this:
If we look at the object-oriented systems, they essentially have abandoned the relational model, as the basis of their object-oriented programming. They’re back to a network model because the relationships they support for the object-oriented programming. All of the data modeling tools are based on the network data model. If you think about the problem from a conceptual point of view, the network model is the survivor simply because it allows you to say more. Or say more, because you’re communicating with somebody else, and you want them to learn more about what you’re doing. The relational model doesn’t suffice for that purpose.
Charles W. Bachman, in an interview by Thomas Haigh
